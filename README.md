# requirements-structuring

一个用于需求梳理的 Codex Skill。它会把对话、会议纪要或零散需求整理成固定六段结构，方便继续评估、确认和推进。

## 输出结构

- 需求描述
- 现状
- 需求评估
- 待确认清单
- 方案说明
- 【价值点】卡点&关键点

## 适用场景

- 需求信息分散，需要先收敛问题定义
- 方案还不明确，需要先识别前提、风险和待确认项
- 想把口头沟通快速整理成可继续协作的结构化简报

## 使用方式

在 Codex 中显式调用：

```text
Use $requirements-structuring to turn the current conversation into a structured requirement brief.
```

也可以直接用自然语言触发，例如：

```text
请根据这段对话，整理成需求描述、现状、需求评估、待确认清单、方案说明、【价值点】卡点&关键点。
```

## 示例

输入：

```text
我们想把异常工单页面改成更适合排查问题的形式，现在字段很多但重点不清楚。希望先帮我梳理需求、现状和可选方案，再列出需要业务确认的问题。
```

输出：

```markdown
### 需求描述
- 需要将异常工单页面从“字段展示导向”调整为“问题排查导向”。

### 现状
- 当前页面字段较多，但信息层级不清，难以快速定位问题。

### 需求评估
- 目标方向明确，但缺少角色、核心排查流程、优先级等关键信息。

### 待确认清单
- 页面主要使用角色是谁。
- 排查时最关键的 3 到 5 个字段是什么。

### 方案说明
- 推荐先基于排查路径重组信息分区，再决定字段取舍和默认展示策略。

### 【价值点】卡点&关键点
- 价值点：降低排查成本，提高定位效率。
- 卡点：缺少核心用户和关键字段优先级输入。
- 关键点：先明确排查路径，再设计页面结构。
```
