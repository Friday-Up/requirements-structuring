# Requirements Structuring

## Overview

将零散信息压缩成固定结构的需求梳理结果，先区分事实、判断、假设，再输出可用于继续沟通或推进方案的摘要。

## Core Workflow

1. 识别输入来源。
   明确内容来自当前对话、文档片段、会议纪要、代码现状或用户补充说明。
2. 抽取已知事实。
   只保留可以从输入直接确认的信息，不要把推测写成既定事实。
3. 识别缺口与风险。
   找出范围不清、目标不清、边界不清、责任不清、依赖不清的部分。
4. 给出结构化整理结果。
   严格按固定栏目输出，帮助用户快速进入确认或决策。

## Output Rules

- 严格按以下顺序输出六个栏目，不要打乱。
- 栏目标题使用中文原文，不要改写。
- 优先写结论性短句，避免空泛描述。
- 明确区分三类信息：
  - 已知事实：输入中能直接确认的内容。
  - 推断判断：基于现状作出的合理分析，需要注意措辞。
  - 待确认项：没有证据支撑、必须由用户或相关方确认的内容。
- 信息不足时，不要硬补；直接在“待确认清单”中指出缺口。
- 如果输入里没有明确方案，也要在“方案说明”里给出建议方向，并说明前提条件。

## Standard Structure

输出时使用以下六段结构：

### 需求描述

- 概括用户真正想解决的问题、目标结果、期望产出。
- 如果用户表达的是动作而不是目标，要补足背后的业务诉求。

### 现状

- 总结当前已知背景、已有做法、现有限制、上下游依赖。
- 写清楚“现在是什么样”，不要混入方案。

### 需求评估

- 评估需求是否清晰、范围是否完整、是否存在歧义或隐含前提。
- 指出实现难点、协作风险、时间风险、信息风险。
- 必要时给出优先级或推进顺序建议。

### 待确认清单

- 罗列必须进一步确认的问题。
- 优先列会影响方案选择、排期、边界和责任归属的问题。
- 若可判断确认对象，直接写出应向谁确认。

### 方案说明

- 给出推荐方案或可选方案。
- 说明方案目的、执行思路、适用前提、影响范围。
- 若有多个方案，先给推荐方案，再简述替代方案与取舍。

### 【价值点】卡点&关键点

- 价值点：说明这项需求或方案能带来的收益、改善或业务价值。
- 卡点：指出当前推进受阻的位置、依赖项、关键风险。
- 关键点：指出最需要被关注、拍板或先行验证的部分。

## Quality Bar

- 优先还原“问题”而不是只复述“要求”。
- 优先指出真正影响推进的问题，不要堆无关问题。
- 优先输出对决策有帮助的信息，不要写成会议纪要流水账。
- 若用户给出的上下文明显矛盾，先指出矛盾，再分别整理。
- 若信息足够清晰，可直接给紧凑版结果；若信息不足，则把缺口写透。

## Style Guidance

- 使用简洁、专业、可执行的表述。
- 保持条目化输出，便于逐条确认。
- 避免口号式表述，例如“提升效率”“优化体验”，除非说明具体体现。
- 避免过度设计，不扩展用户未提及且与当前决策无关的内容。
